## STM32低功耗模式-2020-08-21

### 4.3.3低功耗模式

**进入低功耗模式**

通过执行WFI（wait for interrup）或WFE（wait for event）指令，或者在从ISR返回时将Cortex®-M0+系统控制寄存器中的 SLEEPONEXIT 位置1时，MCU进入低功耗模式。

只有在没有中断待处理或没有事件待处理的情况下，才会通过WFI或WFE的进入低功耗模式


**退出低功耗模式**

MCU进入休眠和停止低功耗模式的方式取决于进入低功耗模式的方式

- 如果使用WFI指令或ISR返回进入低功耗模式，则NVIC确认的任何外设中断都可以唤醒设备。
- 如果使用WFE指令进入低功耗模式，则一旦事件发生，MCU将退出低功耗模式。可以通过以下任一方式生成唤醒事件：
  - NVIC IRQ中断 
 当Cortex®-M0+系统控制寄存器中的 SEVONPEND = 0 时：通过启用外设控制寄存器和NVIC中的中断。当MCU从WFE恢复时，必须清除外围中断挂起位和NVIC外围IRQ通道挂起位（在NVIC中断清除挂起寄存器中）。
 当Cortex®-M0+系统控制寄存器中的 SEVONPEND = 1 时：通过启用外设控制寄存器和NVIC中的中断（可选）。当MCU从WFE恢复时，必须清除外设中断挂起位和使能NVIC外设IRQ通道挂起位（在NVIC中断清除挂起寄存器中），所有NVIC中断都将唤醒MCU，即使禁用的中断也是如此。
  - 事件
  在事件模式下配置EXTI线路。当CPU从WFE恢复时，由于未设置与事件线相对应的挂起位，因此无需清除EXTI外设中断挂起位或NVIC IRQ通道挂起位。外围设备。
 
在外部复位（NRST引脚），IWDG复位，使能的WKUPx引脚之一上的上升沿或下降沿或发生RTC事件时，MCU退出待机和关断低功耗模式。请参见 图280：RTC框图。从待机或关机模式唤醒后，程序执行将以与复位后相同的方式重新启动（引导引脚采样，加载选项字节，读取复位向量等）。


### 7钟低功耗模式


#### 待机模式  Standby mode

VCORE域已关闭电源，但是可以保留SRAM内容：
- 当PWR_CR3寄存器中的RRS位置1时，具有SRAM保留的待机模式。在这种情况下，SRAM由低功耗稳压器提供。
- 待机模式，当PWR_CR3寄存器中的RRS位清零时。在这种情况下，主稳压器和低功率稳压器都将关闭

VCORE域中的所有时钟均停止，并且PLL，HSI16和HSE振荡器被禁用。 LSI和LSE振荡器可以保持运行状态.RTC可以保持活动状态（带RTC的待机模式，不带RTC的待机模式）。退出待机模式时，系统时钟为HSI16振荡器时钟。

#### 关机模式 Shutdown mode

VCORE域已关闭。 VCORE域中的所有时钟都停止，PLL，HSI16，LSI和HSE振荡器被禁用。 LSE可以保持运行。退出关机模式时，系统时钟为HSI16振荡器时钟。

在该模式下，电源电压监测被禁用，并且在电源电压下降的情况下，产品性能无法得到保证。

关机模式可实现最低功耗。它基于深度睡眠模式，禁用了稳压器。因此，VCORE域已关闭电源。 PLL，HSI16，LSI和HSE振荡器也被关闭。除了RTC域中的寄存器外，SRAM和寄存器内容都会丢失。在关机模式下，POR / PDR和BOR不可用。在此模式下无法监视电源电压。结果，当VDD电源丢失时，不支持将RTC域切换到VBAT电源。

**关机模式下的I / O状态**

在关机模式下，可以通过上拉（请参阅PWR_PUCRx寄存器（x = A，B，C，D，F）配置I / O，也可以通过下拉（请参考PWR_PDCRx寄存器（x = A，B，C，D，F）），或可以保持模拟状态，但是由于上电复位而退出关机模式时，此配置会丢失。

PC13上的RTC输出可在关机模式下工作。 LSE所用的PC15和PC15都可以使用，5个唤醒引脚（WKUPx，x = 1,2,4,5,6）和两个 tampers 可用

**进入关机模式**

当Cortex®-M0+系统控制寄存器中的 SLEEPDEEP 位置1时，MCU根据 `进入 低功耗模式` 部分进入关断模式。  
有关如何进入关断模式的详细信息，请参见表28：关断模式摘要。  
在关机模式下，可以通过编程各个控制位来选择以下功能
- 实时时钟（RTC）和篡改（TAMP）：这是由RTC域控制寄存器（RCC_BDCR）中的RTCEN位配置的。
- 外部32.768 kHz振荡器（LSE）：由RTC域控制寄存器（RCC_BDCR）中的LSEON位配置

>注意：当VDD在关机模式下掉电时，RTC域内容丢失。

**退出关机模式**

MCU根据 `退出低功耗模式` 部分退出关机模式。从关机模式退出时会发生上电复位。从关机唤醒后，所有寄存器（RTC域中的寄存器除外）均复位。有关如何退出关机模式的更多详细信息，请参见表28：关机模式摘要。

## GPIO_Analog 引脚模式-STM32CubeMX-2019-05-17

最近开始学习使用STM最新的编译器STM32cubeIDE，直接从Keil转移过来的。之前也没用过CubeMX。

这里在配置引脚模式时，遇到了一个问题：**GPIO_Analog** 是什么模式？之前似乎从没遇见过。


![enter description here](https://LonlyPan.github.io/images/Posts/2019-05-17-GPIO_Analog_引脚模式-STM32CubeMX/引脚选择.png)

上网搜了一通，找到ST官方社区的两个问答，才解决了疑惑。

问答原址：  

[1、Question about ADC versus GPIO Analog](https://community.st.com/s/question/0D50X00009XkfqtSAB/question-about-adc-versus-gpio-analog)  
[2、What pins can I use for Analog Input/Output? STM32CubeMX allows every GPIO to be set to ''GPIO_Analog''?](https://community.st.com/s/question/0D50X00009XkWkeSAF/what-pins-can-i-use-for-analog-inputoutput-stm32cubemx-allows-every-gpio-to-be-set-to-gpioanalog)  

总结起来就两个：

1、**GPIO_Analog**和**A/D**没有啥关系，就是名字是这个（第一个问答后面就一直在探讨两者关系，最终的答案是：当开启ADC功能时，后台默认同时开启Analog功能。但开启Analog功能，就是Analog功能。但笔者的理解：两者其实就没有联系）。  
2、这个功能实际作用是**降低功耗**。通过关闭引脚的上拉、下拉和施密特触发器，减小电流损耗。

实际STM32中文参考手册的**GPIO功能描述**也有介绍，不过这是个坑。我们看一下对比。

![参考手册对比](https://LonlyPan.github.io/images/Posts/2019-05-17-GPIO_Analog_引脚模式-STM32CubeMX/20190517参考手册对比.png)

官方的描述是**Analog**，翻译过来就是**模拟输入**了。当阅读的时候直接照字面理解成立类似ADC的模拟输入引脚了，但实际不是这个意思。当然中文版是**V10**，我的这篇英文是**V20**版，存在改版可能性。好了，看一下手册上对**Analog**模式的解释吧。

## 结构体使用前必须分配内存-2019-5-18

最近再将原Keil5中的STM32工程移植到Atollic TrueSTDIO for STM32上，顺便使用STM32CubeMX简化初始化配置，减小工作量。更具体的原因则是涉及到版权和商业化问题，另外Keil所有版本的破解版都是2020年到期，我很慌啊。。。

但移植就以为这一堆问题，工程文件配置什么的都好解决，今天遇到的是程序上的问题。那就是：结构体使用前必须分配内存

原Keil函数里使用用到结构体的，但我在一个函数中申明了它却没有申请内存，关建是编译没有错误和警告，而且实机测试也是成功的。而今天原模原样照搬到TrueSTDIO上，编译也是没有错误和警告的，但实机测试情况是能跑，但不正常。最后一点点在线调试，才发现是结构体的问题，遂申请内存空间，问题解决。最后为了解答心中疑惑，我又测试了一下：

### TrueSTDIO调试，出错输出信息：

硬故障HardFault

硬故障详细信息
总线，存储器管理或使用失败（FORCED）

总线故障详细信息
 不精确的数据访问冲突（IMPRECISERR）

MSP = 主堆栈指针
PSP = 进程堆栈指针

### 编译测试

在线代码编译：（经常用来跑测试）
https://www.onlinegdb.com/

测试代码：
```c
#include <stdio.h>

typedef struct{
	__uint16_t a;    // 随意的变量
	__uint16_t b;
} test;

int main() {
    test * test1 = NULL; 
    //test1=(test*)malloc(sizeof(test));//分配内存
    test1->b=5; 
    printf("b = %d",test1->b); 
    return 0;
}
```
上面将内存分配注释掉，运行时直接报错编译不通过，输出结果：`Segmentation fault (core dumped)`

去掉注释后，就正常了，输出结果：`b = 5`

同时我也在C++环境下测试，同样的结果。

### 结论

至此得出结论：**结构体在使用时是必须要分配内存的；Keil 5在编译时，可能对结构体会有优化，使得不分配内存也不会出错。**

## IO部分引脚不工作

### 测试环境

- STM32G031G8 芯片
- SWD下载接口
- Micro-USB 硬件接口（SWD接在数据引脚上）

### 描述

micro-usb 为手工焊接，导致接触不良，使得程序调试下载时，一会可以一会报错（无法下载）。但正常下载后，发现部分引脚无输出（测试程序令所有引脚输出高电平），部分有输出。检查了硬件设计没有问题，程序也没问题。重新焊接了 MCU，问题依旧未解决。遂根据下载不正常判断是不是 usb 焊接问题。重新焊接，下载程序不再故障，IO 输出也都正常。

### 结论

一开始下载不正常时就应该判断出 usb 没焊好，但心存侥幸、懒。没去在意和解决。  
事出离奇必有妖。不能容得半点马虎。  
至于为何接触不良会导致程序执行不正常，猜测是下载数据传输错误。具体原因无法深究查明。

## 不同型号单片机SPI通信时序不同

采用STM32F103RCT6，设置为单向主模式，通信时序如下：
![enter description here](https://LonlyPan.github.io/images/Posts/2020-10-09-STM32_调试-错误笔记/20201215153040.png)
采用STM32F401CCU6，设置为单向主模式，通信时序如下：
![enter description here](https://LonlyPan.github.io/images/Posts/2020-10-09-STM32_调试-错误笔记/20201215152923.png)

橙色是时钟 SCK，蓝色是数据发送 MOSI。其中红框部分为一个主程序循环，每个循环发送两次数据。可以看到 103 的时序似乎有些问题，其中一个数据的发送看似是平的，而 401 的则正常。但两者都可以驱动设备正常工作（屏幕显示）。一开始怀疑是硬件问题或者底层配置问题，纠结一天没有解决。然后，当我把这两个图放在一起对比时，忽然明白：那个看似时平的数据时序其实也是在发送数据，不过他的数据全是1。查看程序，果然，103 驱动屏幕显示的颜色在白色和红色之间切换，而白色对应的数据为 0xFFFF，导致波形数据是平的。401的则是在蓝色和红色之间切换。

所以这并不是问题，只是再次提醒自己，遇到问题应冷静分析，使用数据说话，而不是在哪瞎猜或者瞎尝试。

### 解决方案

一开始我是各种搜索问题答案，但是资料是在太少，且基本是没有解决问题。自己整了2小时，才解决。

问题竟是出在杜邦线上，我是使用杜邦线连接J-Link和设备的，且下载速率设为了4MHz。这就导致两个问题：
- 杜邦线接触不良，会导致**错误1**或**错误2**
- 杜邦线下载速率太高，会导致**错误2**或**错误3**

**解决方法：**
- 更换杜邦线，选择接头紧的，不要出现松动
- 降低下载速率，1MHz以下
